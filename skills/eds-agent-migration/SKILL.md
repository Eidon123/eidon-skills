# EDS Agent Migration

把一次跨 Agent 工作台迁移做成可审计、可回滚、可长期维护的结构变更。

本 skill 负责：

- 确认迁移目标、目标宿主、规则发现机制和完成标准。
- 迁移跨宿主公共规则与宿主专属适配层。
- 识别规则层和 skill 层之间的依赖，编排整体顺序。
- 调用 `eds-skill-manager` 处理 skill 归拢、更新、`SOURCE_OF_TRUTH.md`、目录清单和入口健康。
- 对规则层与 skill 层的组合结果做最终验证。

本 skill 不负责日常 skill 安装、更新、卸载、修复、目录维护或入口治理。没有规则文件或整体工作台迁移目标时，把 skill 层工作交给 `eds-skill-manager`。

## 不变量

1. 先审计，后判断；先确认，后写入。
2. 不把复制规则文件当作迁移，不把发现路径当作内容真源。
3. 不根据宿主名称硬编码扫描路径、规则文件名、bridge 字段或触发方式。
4. 不因同名、跨项目出现或单宿主存在就直接合并、全局化或删除。
5. 不覆盖、回退或清理用户已有改动；遇到重叠修改时先停下来说明。
6. 不把 manager 的日常维护职责复制进本 skill。
7. manager 不可用时允许完成独立的规则层工作，但不得伪造完整 skill 迁移。

## 四层身份模型

Skill 层使用以下模型交换证据；Migration 只标记候选路径和角色线索，逐 skill 的最终身份清单由 manager 产出：

1. **作者真源**：长期编辑正文和运行资产的位置，可能是项目目录或独立 Git 仓库。
2. **发布上游**：GitHub、插件市场、well-known、包管理器或其他分发来源；本地自用 skill 可以没有此层。
3. **标准安装副本**：安装器或同步流程生成、供本机运行的标准副本；它可能含生成元数据，不等于作者真源。
4. **宿主发现入口**：宿主实际扫描到的目录、软链接、bridge、插件生成入口或逻辑别名。

允许多层指向同一路径，也允许某层不存在，但必须把角色写清。遇到软链接时同时记录逻辑路径与 `realpath` 后的物理路径。不要用“真源”一词同时指代作者目录、安装目录和宿主入口。

规则文件也要区分：

- 公共规则真源。
- 宿主专属适配层。
- 宿主实际加载入口。
- 由 manager 维护的结构契约 `SOURCE_OF_TRUTH.md`。

## 与 eds-skill-manager 的边界

### Migration 负责

- 跨宿主目标和迁移边界。
- 各宿主规则文件的发现、优先级和冲突审计。
- 公共规则提炼、宿主适配层设计与规则文件写入。
- 把规则层结论传给 manager。
- 统一写入确认、执行节奏、组合验证和收尾结论。

### Manager 负责

- 项目与全局 skill 的逐项盘点和四层身份登记。
- skill 归拢、命名、frontmatter、安装、更新、卸载和修复。
- 标准安装目录、外部托管套件、单宿主例外和手工同步项的分类。
- lock、catalog、upstream、版本与成员策略。
- `SOURCE_OF_TRUTH.md` 的创建和维护，包括接收 migration 提供的规则层映射。
- 宿主发现入口、软链接、bridge、重复项和悬空项的健康维护。

### 联动协议

在 skill 层需要变化时，加载 `eds-skill-manager`，向它提供：

- 本轮目标宿主和完成标准。
- 已验证的规则与 skill 发现矩阵。
- 候选项目目录、全局目录、宿主目录和 upstream 路径。
- 规则层拟定的公共真源、适配层和入口映射。
- 用户批准的操作边界、保留项、删除项与外部副作用。
- dirty worktree、备份位置和不可触碰路径。

要求 manager 返回：

- skill 四层身份清单和不确定项。
- 精确的创建、修改、移动、覆盖、删除、保留清单。
- `SOURCE_OF_TRUTH.md` 与入口健康处理方案。
- 执行结果和可复核的验证证据。

Migration 保留最终编排与确认责任。Manager 先返回精确预检 manifest；Migration 向用户统一确认后，把完全相同的已批准 manifest 作为执行授权传回 Manager，避免重复询问。Manager 返回的范围、风险或外部副作用扩大时，Migration 必须重新确认。

### Manager 不可用时

如果找不到、无法读取或无法调用 `eds-skill-manager`：

1. 继续完成只读规则审计和只读 skill 概览。
2. 明确列出 manager 缺失导致无法安全完成的部分。
3. 只有当规则层能与 skill 层解耦且用户确认接受部分迁移时，才执行规则层写入。
4. 不移动、合并、删除、更新或重命名 skill。
5. 不新建或改写 `SOURCE_OF_TRUTH.md`，不修复宿主 skill 入口。
6. 收尾标记为“规则层完成，skill 层未完成”，不得称为完整迁移。

## 工作流

### Phase 0：锁定目标

先确认或从环境中推导：

- 本轮是纯审计、单向迁移、反向补齐、多宿主对齐，还是混乱工作台整理。
- 目标宿主与不在范围内的宿主。
- 项目级、全局级或两者是否都在范围内。
- 允许的动作：只读、写规则、联动 manager、移动、删除、联网更新、发布。
- 完成标准和不能中断的既有使用方式。

能通过只读检查确定的事实不要询问用户。仍会改变架构或造成外部副作用的选择必须询问。

### Phase 1：建立实际发现矩阵

对每个目标宿主分别验证规则与 skill 的发现机制。优先证据顺序：

1. 当前安装版本的本地官方说明、配置或可读实现。
2. 宿主提供的只读列表、诊断或 inspect 命令。
3. 已存在入口的真实指向和可复现加载结果。
4. 官方文档。
5. 用户确认。

为每条发现关系标记：`已验证`、`有证据推定` 或 `未知`。至少记录：

- 宿主名称和版本。
- 项目级、全局级规则加载路径及优先级。
- 项目级、全局级 skill 扫描路径。
- 是否递归扫描、是否跟随软链接、是否读取通用目录。
- 插件或官方安装器是否生成特殊入口或命名。

不要从目录存在反推宿主一定扫描它。不要从一个宿主的行为推断另一个宿主。若必须用临时测试 skill 验证，先单独说明写入和清理动作并取得确认；测试后立即清理并复核。

### Phase 2：只读审计

#### 规则层

检查所有目标宿主实际加载的规则文件及其引用，记录：

- 路径、作用域、优先级和宿主。
- 公共内容、宿主专属内容、重复内容和互相冲突的内容。
- 悬空引用、失效路径和宿主不识别的写法。
- 哪个文件当前被长期维护，哪个只是复制或适配层。

确认公共规则脱离任一单宿主后仍能成立。遇到语义冲突时不要自行选择较弱规则；列出影响并请求决定。

#### Skill 层

只收集 manager 启动所需的边界证据：

- 候选作者真源、发布上游、标准安装副本和宿主入口。
- 目录、文件、软链接、Git 仓库、插件目录和安装器痕迹。
- 同名、同 hash、分叉、悬空和单宿主项。
- 相关 lock、catalog、版本文件和更新工具。

不要在 migration 中完成 manager 的逐 skill 维护判断。

#### 工作区状态

对所有可能写入的 Git 仓库读取：

- 当前分支、HEAD、remote。
- `git status` 和未提交改动。
- 本轮拟修改文件是否已被用户改动。

对非 Git 路径记录文件类型、权限、软链接目标、hash 或可复核元数据。发现不属于本轮的改动时保留并避开；发现同文件重叠改动时说明合并策略，无法安全合并就停止。

### Phase 3：设计目标结构

先设计规则层：

- 选择一个公共规则真源；只有实际宿主支持时才直接复用该文件。
- 把宿主专属命令、字段和限制保留在薄适配层。
- 宿主无法引用公共真源时，明确采用何种受控副本或生成机制及其漂移检查。
- 记录规则优先级，避免公共层和适配层互相覆盖或循环引用。

再让 manager 设计 skill 层与 `SOURCE_OF_TRUTH.md`。入口只在宿主无法直接发现目标内容时创建。整目录软链接、逐项软链接、thin bridge、官方生成入口或无需入口都必须基于 Phase 1 的证据选择。

形成一个主迁移方案，至少包含：

- 迁移后的规则关系和四层 skill 关系。
- 保留的官方机制、合理单宿主例外和外部托管套件。
- migration 与 manager 各自执行的动作。
- 备份、回滚、风险和验证方式。
- 尚未解决且会影响执行的决策。

全局唯一 `SOURCE_OF_TRUTH.md` 由 manager 维护。Migration 提供公共规则、宿主适配层、加载优先级和入口映射；SOT 只记录约定与非标准例外，不复制各项目规则正文。

不要用固定多方案模板代替判断。只有存在真实高风险分歧时才列备选。

### Phase 4：写入前闸门

任何写入前向用户展示精确清单：

- 创建。
- 修改。
- 移动或重命名。
- 覆盖。
- 删除。
- 保留且不触碰。
- 将运行的安装、更新、发布或其他外部副作用命令。
- migration 与 manager 的责任分工。
- 备份位置、恢复步骤和验证标准。

明确指出 dirty worktree 中哪些既有改动会被原样保留。取得明确确认后才执行。确认只覆盖已列动作；执行中发现新路径、新删除项、新联网动作或更大范围时，暂停并重新确认。

### Phase 5：备份

在第一项写入前创建时间戳备份和清单：

- 只备份会被修改、移动、覆盖或删除的路径。
- 备份到宿主扫描目录之外。
- 软链接备份链接本身及其目标说明，不要无意递归复制目标。
- 保存变更前路径、类型、链接目标、hash、权限和 Git 状态。
- 验证备份可读，并给出最小恢复步骤。

备份失败时停止，不继续写入。Git 历史不能代替未提交文件、全局目录和软链接的备份。

### Phase 6：执行并持续汇报

按以下顺序执行已批准动作：

1. 写入公共规则真源。
2. 写入或修正宿主适配层与规则入口。
3. 重新验证规则层，再把最终映射交给 manager。
4. 让 manager 执行已批准的 skill、`SOURCE_OF_TRUTH.md` 和入口动作。
5. 执行组合验证。

执行期间持续发送进度，不要长时间静默：

- 每个 Phase 开始和结束时汇报一次。
- 长扫描或批量处理时，每完成一批或最长约 60 秒汇报一次。
- 只汇报当前动作、已确认事实、下一步和阻塞，不输出隐藏推理过程。
- 发现风险、范围变化或验证失败时立即汇报并暂停相关写入。

不要一次性做完后才说明。不要把“命令成功”当作“迁移成功”。

### Phase 7：最终验证

#### 规则层验证

- 每个目标宿主实际加载到了预期公共规则和适配层。
- 公共规则可以独立理解，适配层只含必要差异。
- 优先级、引用和路径没有循环、悬空或冲突。
- 未把宿主专属语法误写进其他宿主的公共层。

#### Skill 层验证

复核 manager 的证据：

- 每个纳入范围的 skill 四层身份明确，缺失层有合理说明。
- 目录名、frontmatter `name`、安装清单和入口名称符合既定策略。
- 所有目标 `SKILL.md` 可读；需要一致的副本 hash 一致。
- 软链接和 bridge 指向预期物理路径，没有悬空、循环或意外分叉。
- 每个目标宿主通过已验证的发现机制看见预期集合，不存在为“看起来一致”而创建的冗余入口。
- upstream、lock、catalog、手工同步项、外部托管套件和成员策略彼此一致。
- `SOURCE_OF_TRUTH.md` 记录规则层、四层身份、宿主差异、例外、删除项和维护流程。

#### 变更边界验证

- 对照变更前清单，确认实际改动没有超出批准范围。
- unrelated dirty changes 仍在且内容未被回退。
- 备份仍可读，恢复说明与当前路径对应。
- 执行必要的宿主发现冒烟测试；无法直接验证的项目明确标记为未验证。

任何关键验证失败都要修复后重验；修复超出确认范围时先重新确认。不要以部分通过冒充完整通过。

按证据分级收尾：所有目标宿主完成真实加载与发现冒烟测试才标记为 `完整迁移`；仅验证规则和发现机制、尚未启动目标宿主时标记为 `结构可运行，宿主待实测`；仍有未知发现机制或 manager 未完成时标记为 `部分迁移/仅审计`。

## 收尾

用简洁结果说明：

- 本轮是完整迁移、规则层完成但 skill 层未完成，还是仅完成审计。
- 最终公共规则真源、各宿主适配层和加载关系。
- manager 确认的 skill 四层结构、`SOURCE_OF_TRUTH.md` 与入口状态。
- 保留的例外、未验证项和剩余风险。
- 备份位置和恢复入口。
- 后续维护边界：规则迁移继续使用本 skill；日常 skill 与入口维护使用 `eds-skill-manager`。

## 禁止模式

- 不硬编码 Claude Code、Codex、Grok 或其他宿主的发现路径和专属 frontmatter。
- 不把当前机器上的数量、目录名或某个案例写成通用架构要求。
- 不把逻辑发现路径直接当作物理安装根；先解析软链接。
- 不编辑安装副本来替代已有作者真源。
- 不把多 upstream 的标准安装目录初始化为聚合 Git 仓库。
- 不把项目 skill 因跨宿主使用而自动变成全局 skill。
- 不把官方管理的外部套件强行复制进标准安装目录。
- 不为追求宿主集合表面一致而覆盖官方命名或生成策略。
- 不在未确认时写入、移动、覆盖、删除、联网更新或发布。
- 不在 manager 不可用时自行接管其维护职责。
- 不维护冗长 bridge 模板；根据已验证机制生成最小必要入口。
